# 版本历史

本文档记录了 BaiduPCS-Rust 的所有版本更新历史。

## v2.1.1 (当前版本)

**问题修复：**
- 🐛 **扫码登录**：确认环节失败/无真实 uid 时不再降级成 `uid=0` 的「临时用户」假成功，改为如实返回失败提示重新扫码——修复假登录导致后续网盘 API 全部 `errno=-6`
- 🐛 **系统设置**：左侧锚点导航点击不再把顶部「保存设置」栏顶出可视区（只滚动内容容器，不再连带滚动外层）
- 🐛 **分享同步**（后端修复与增强均来自 @hamr-hub PR #111；其中孤儿 run、`resume_link_invalid` 状态不一致为下游补充修复）：
  - 手动触发：触发前做前置校验并返回真实 `run_id`，不再「假成功」
  - 调度器：`on_tick` 改为返回 `Result` 不再静默吞错，失败落 `warn` 日志便于排查
  - 下载等待：改为「空闲超时 + 硬上限」双闸，文件夹聚合速度改用「活跃下载状态」判断，修复进行中速度恒显示 0
  - 历史数据：启动期对 `owner_uid=0` 历史脏数据按活跃账号迁移归属
  - （下游补充）修复为「立即返回 run_id」预建 run 引入的孤儿 run（早退路径未收尾导致永久 `running`）与 `resume_link_invalid` 的状态不一致

**新功能：**
- ✨ **分享同步**（感谢 @hamr-hub PR #111）：
  - 运行明细分页：新增 `GET /share-sync/runs/:id/items` 分页接口（`count` + 分页查询），前端运行详情弹窗接入分页，避免大 run 明细被截断到 100 条
  - 下载停滞自动重试：检测到下载长时间无进展时自动 `pause→冷却→resume` 重启（限次数 + 冷却，可经环境变量调参）

**工程/CI：**
- 🔧 统一 Rust 工具链版本为 1.87（本地 / Docker / CI 一致）
- 🔧 增加 Conventional Commits 校验（PR 标题 + 逐 commit）

## v2.1.0

**新功能：**
- ✨ **分享同步（Share Sync）**：订阅第三方分享链接，按固定间隔或每日定时拉取文件列表，与上次快照对比差异，自动转存到网盘和/或下载到本地（感谢 @hamr-hub PR #98、#99、#102）
  - 多账号隔离（按 `owner_uid` 归属与解析），单条订阅可同时配置「网盘 + 本地」双目标，支持 include/exclude 路径过滤
  - 三种冲突策略（覆盖 / 新版本 / 跳过），删除行为可控（默认保留旧文件，可选「分享者删除 → 目标也删除」）
  - 实时子任务进度（速度 / 预计剩余时间）、运行历史审计（added/modified/removed/failed + 文件级 run_item），WebSocket 实时推送 + REST 轮询兜底
  - Web UI：`/share-sync` 订阅列表 + 创建（复用转存对话框）/ 运行历史弹窗 + 内联子任务进度

**问题修复：**
- 🐛 **分享同步·增量同步**：已同步目录里改动/新增单个文件时只同步变动的文件，不再整目录重转存 + 重下载
- 🐛 **分享同步·转存冲突**：转存撞同名不再整批判失败，视为已存在并继续下载；修复增量单文件误判失败与网盘目录名翻倍（`/测试/测试/...`）
- 🐛 **分享同步·状态**：进行中子任务列表不再回包已完成的文件夹；重启后自动续跑被中断的 run（不再粗暴标失败）；删除订阅时清理名下内部转存/下载任务，避免孤儿脏数据
- 🐛 **风控**：列目录抓快照/拆批接入全局限速器；删除大目录、整批转存自动分块规避 `errno=132`，整批转存临时超时（`errno=4`）退避重试；轮询抖动不低于最小间隔；链接连续失效自动暂停轮询 + 前端「恢复」
- 🐛 **进度显示**：修复文件夹下载已下载翻倍（200%）与上传超过 100%（142%）——已上传改用「已完成分片之和」（幂等），全站进度条与后端 `progress()` 钳到 [0,100]
- 🐛 **转存**：重启恢复转存任务时还原 `is_internal`，分享同步内部转存任务不再漏进转存管理
- 🐛 **Windows 控制台日志乱码**：Windows 下默认关闭控制台 ANSI 彩色输出，避免出现 `[2m`、`[32m` 等转义字符；非 Windows 平台保留终端彩色，文件日志始终为纯文本（感谢 @YYQHH PR #103）
- 🐛 **前端**：文件列表滚动加载在请求失败后不再无限翻页；修复路由切换后 el-table 列宽错位；修复 `npm run lint`（ESLint v9 flat config）并清理零风险静态告警

## v2.0.0

> 🎉 重大版本：引入**多账号支持**。版本号从 1.14.1 跃升至 2.0.0——多账号涉及任务归属、数据隔离与资源调度的架构级改动，按语义化版本属于不兼容的大版本升级。

> ⚠️ **升级前请先备份后端数据目录**：`backend/config/`（含 `baidu-pcs.db`、`accounts.json`、`session.json`、`autobackup_configs.json`）和 `backend/wal/`（任务状态 `.meta`）。Docker 用户备份挂载的 `data` 卷即可。万一异常，恢复这两个目录即可回到升级前状态。
>
> ⚠️ **降级提示**：如需回退到旧版本，请**先恢复升级前备份的 `config/` 与 `wal/` 目录，再启动旧版本**；不建议直接用 2.0.0 升级后的数据目录启动旧版本。原因：2.0.0 会改动数据（SQLite 新增 `owner_uid` 列、`.meta` / 历史回填、`session.json` 改名迁移），旧版本对这些改动的兼容性无法保证；且旧版本为单账号，会把多账号历史混在一起显示。若已无升级前备份，回退后至少需要**重新登录**。
>
> 📌 **自定义 DB 路径用户注意**：若你在 `config/app.toml` 的 `[persistence].db_path` 把数据库放到了 `config/` 之外，则备份 / 还原 `config/` 与 `wal/` **并不包含该数据库文件**——请额外备份 / 还原该真实 DB 文件（含其 `-wal` / `-shm` 边车文件）。系统的自动迁移前备份会按 `db_path` 复制真实 DB 到 `config/backups/pre_migration_<时间戳>/`，但**手动恢复时需把它放回原自定义位置**。

**新功能：多账号**
- ✨ **多账号独立任务管理**：可添加 / 切换 / 删除多个百度账号；每账号独立客户端实例（BDUSS / Cookie / 会话），下载 / 上传 / 转存 / 自动备份 / 离线下载任务均按 `owner_uid` 归属与隔离
  - 持久化数据迁移（单账号 → 多账号）；删除账号按 `owner_uid` 精确清理本地任务与历史
  - 跨账号聚合列表按 id 去重；历史回退路由按 `owner_uid` + `task_type` 精确匹配，避免跨账号 / 跨类型误删
  - 修复账号切换 / 操作备份任务后的死锁与父任务永久卡 `Transferring`；统一收口 uploader 跨 `await` 持 DashMap 锁，消除多账号并发下的死锁
- ✨ **账号切换器与账号管理面板**：顶部一键切换账号，设置页集中管理账号
- ✨ **实时资源配额（BudgetScheduler）**：机器总线程 M 按会员权重（普通:VIP:SVIP = 1:3:5）在账号间分配，每账号实时展示「保底 base / 上限 vip_cap」两条水位
  - 「自动按等级推荐」按 VIP 等级取上限（普通会员 10 / 超级会员 15），并受机器总线程与权重共同约束；资源配额面板显示真实生效值与实际可用线程
  - 优先级信号量（priority semaphore）公平调度各账号的分片并发
- ✨ **数据安全兜底·全局只读模式**：启动期「单账号 → 多账号」迁移或关键持久化失败时，自动进入**全局只读**（拦截所有写操作、保留读 / 历史，前端提示进入保护模式），避免数据不一致继续扩散；修复后端问题后重启即恢复
- ✨ **断线自适应退避重连**：后端不可达时下载 / 上传页按阶梯退避重试，0 B/s 显示「计算中」，避免临时失败导致列表清空与轮询停止

**问题修复（与多账号无关的通用项）：**
- 🐛 下载取链 `errno=8002` 临时风控时阶梯式重试；`filemetas` 按路径查询改用 `xpan/file list` 解析 `fs_id`
- 🐛 创建即持久化下载任务，重启后保留终态失败任务

## v1.14.0

**新功能：**
- ✨ **文件管理：重命名 / 批量复制 / 批量移动**：文件管理页面新增文件、文件夹的重命名以及批量复制 / 批量移动到目标目录
  - 调用百度 `filemanager` 接口异步模式（`async=2`），后端 `taskquery` 长轮询直到任务终态
  - 文件名校验：拦截 `\ / : * ? " < > |`、控制字符、Windows 保留名（CON/PRN/...）等非法输入
  - 适配百度风控：`errno=132` / `authwidget` / `verify_scene` 透传到前端，日志中对 `authwidget.safesign` / `safetpl` / `saferand` 字段脱敏
- ✨ **HTTP/2 → HTTP/1.1 自适应降级**：连续 3 次零字节失败（含 H2 frame error / 连接断流 / SSL 握手失败等）时，全局下载客户端自动切到 HTTP/1.1 并保持 10 分钟，到期后台任务自动重建为 HTTP/2 客户端，避免 H2 协议故障下整个下载链路长时间不可用
- ✨ **分片失败重试策略精细化**：分片下载失败按 `Transient`（瞬时抖动）/ `ServerThrottled`（429/503 限流）/ `Fatal`（链接全挂）/ `Cancelled`（用户取消）四类分别处理，分别走 100ms→5s 指数退避、1s→10s 加长退避、`mark_deferred` 等待其他分片完成、短路返回四条路径
  - `ChunkManager::fail_chunk` 在同一把锁内原子完成 `unmark + retries+=1 + cooldown/deferred` 写入
  - `Cancelled` 路径不计入零字节降级计数、不计入连续失败计数、不走 cooldown/deferred，确保用户取消不会被误判为协议故障而触发 HTTP/1.1 降级
- ✨ **任务级 auto_requeue + 退回冷却**：连续 chunk 失败累计达阈值（`CONSECUTIVE_CHUNK_FAILURE_LIMIT=6`）或调度器检测到「真死锁」（活跃分片为 0 且无冷却分片）时，主动将任务退回等待队列，冷却 `REQUEUE_COOLDOWN_SECS=60` 秒后再试，避免任务死等 deferred 解冻
- ✨ **解密协程 epoch 并发保护**：新增 `decrypt_epoch` 版本号 + `decrypt_committed` 原子提交标志，覆盖「暂停 Decrypting → 快速恢复 → 新一轮解密协程启动 → 旧协程跑完」的 race
  - 旧协程在每个破坏性副作用之前（spawn_blocking 前 / 后 / 进度回调 / 完成处理入口）比对 epoch，过期即跳过 rename / 删除加密文件 / 改写 `task.local_path` / 持久化 / 终态写入
  - 闭合 rename 成功 → 持久化之间的最后时序窗口
- ✨ **退回原因透传到前端**：`DownloadStatusChanged` 事件新增 `error: Option<String>` 字段（`#[serde(skip_serializing_if = "Option::is_none")]`），`auto_requeue_task` 携带退回原因，下载管理页面可直接看到任务为何被退回等待队列
- ✨ **文件夹固定槽位**：新增 `fixed_slot_subtask` / `uses_folder_fixed_slot`，文件夹子任务可独占父文件夹的固定槽位，避免「`slot_id=None` → 申请槽位 → 立即又被打回等待队列」的自循环

**问题修复：**
- 🐛 **修复备份任务删除后父任务卡死**：`UploadManager::delete_task` / `delete_task_internal` 路径下，`is_backup=true` 的任务被 `publish_event` 直接跳过，`AutoBackupManager` 看不到上传子任务被删除，父备份任务的 `pending_upload_task_ids` / `transfer_task_map` 残留，状态机一直卡在 `Transferring` / `WaitingTransfer`。现在删除路径主动补发 `BackupTransferNotification::Deleted`，`AutoBackupManager` 复用 `handle_transfer_completed(success=false)` 路径完整清理
